PostgreSQL Internals

Process-per-connection · MVCC with heap dead tuples · Physical WAL replication

WAL — Physical Replication

WAL operates at the storage page level. PostgreSQL stores data in 8KB pages. When you do:

UPDATE users SET name = 'Alice' WHERE id = 42;

WAL doesn't log "set name to Alice for id 42". It logs:

• The full 8KB data page image (or a diff of it) for the heap tuple
• Page changes for every index on that table (B-tree leaf page modifications) — 5 indexes means 5 more page-level changes
• TOAST table changes if the row has large values
• Visibility map updates (for MVCC)
• Free space map updates

A single logical row change can produce tens of kilobytes of WAL because it's recording the physical byte changes across multiple pages.

Why full page images? After a crash, PostgreSQL needs to recover to a consistent state at the page level. It can't just "replay the UPDATE" because it doesn't know if the page was half-written to disk (torn page). So on the first modification of each page after a checkpoint, WAL writes the entire 8KB page (Full Page Write / FPW). Safe, but extremely verbose.

Fan-Out: Why 50 Replicas Kills the Primary

Primary serializes and ships the bloated WAL stream 50 times. If WAL generation is 200MB/s, that's ~10GB/s outbound from the primary. Each wal_sender process copies and pushes full page images.

CPU Cost of WAL Senders

Each wal_sender process on the primary:

• Copies WAL data from shared buffers or disk
• Handles the replication protocol (keepalives, feedback, flow control)
• For synchronous replication (to the standby), blocks the committing transaction until the standby acknowledges

At 50 replicas, that's 50 OS processes competing for CPU time on the same machine that's already handling all write workload.

The fundamental tension: The very mechanism that makes failover reliable (physical WAL) is the one that chokes the primary when you have 50 replicas hanging off it. This is why OpenAI flagged WAL as a scaling concern.

max_wal_senders — The Connection Ceiling

Each replica streaming from the primary consumes a wal_sender process — a full forked PostgreSQL backend. Controlled by max_wal_senders (separate from max_connections), so they don't block user connection slots. But each one consumes:

• Its own memory allocation (work_mem, process-level buffers)
• CPU time for reading WAL segments and streaming them over the network
• A file descriptor for the replication TCP socket
• Shared buffer access for reading WAL pages from pg_wal

The max_wal_senders GUC defaults to 10 (since PG 10). With 50 replicas, OpenAI would need max_wal_senders = 60 to leave headroom for pg_basebackup, backup tools (Barman), and monitoring tools that read WAL. Each slot pre-allocates shared memory at startup.

Cascading Replication — Available Since PG 9.2

Instead of every replica connecting directly to the primary, you can chain them:

┌─ Replica 3 ┌─ Standby A ─┼─ Replica 4 │ └─ Replica 5 Primary ─┤ │ ┌─ Replica 6 └─ Standby B ─┼─ Replica 7 └─ Replica 8

Primary only feeds 2–3 intermediate standbys — fan-out drops from ~10 GB/s to ~600 MB/s. OpenAI has flagged this as future development because it adds complexity: replication lag compounds per hop, intermediate standby failover needs automation, and there's no built-in orchestration for topology changes.

Why Not Logical Replication for Standbys?

PostgreSQL has logical replication (since PG 10), but it can't replicate DDL, can't replicate sequences or large objects, has higher apply latency, and doesn't support failover. Physical WAL guarantees a byte-identical copy — exactness at the cost of verbosity.

Connections — Process-Per-Connection Architecture

PostgreSQL uses a process-per-connection model inherited from its 1990s Postgres95 roots. Every client connection forks a dedicated OS process. Each backend gets:

• Its own process memory — work_mem (default 4MB, but can be much more per sort/hash), maintenance memory, temp buffers
• A private copy of the catalog cache — parsed table schemas, type info, function bodies
• A PGPROC slot in shared memory — tracked in ProcArray for MVCC visibility checks
• An OS-level process — context switching, TLB flushes, scheduling overhead

At 5,000 connections (OpenAI's scale), you have 5,000 OS processes, each consuming 5–50MB of private memory.

Why High Connection Counts Kill Performance

1. ProcArray contention: Every transaction start/end scans ProcArray (all active backends) for MVCC snapshots. O(n) at 5,000 connections.

2. Context switching: Linux scheduler wasn't designed for 5,000+ CPU-bound processes. Cache lines (L1/L2/L3) thrashed on every switch.

3. Memory bloat: 5,000 backends × 10MB = 50GB private memory before shared_buffers.

4. Shared buffer contention: Hot pages (like an index root) become lightweight lock bottlenecks.

5. File descriptor explosion: Each backend opens its own FDs (not shared across processes). At 5,000 backends caching ~200 FDs each → ~1,000,000 open FDs. Requires ulimit -n 10,000+, kernel fs.file-max in millions, and fs.nr_open adjustments. Every FD consumes a kernel struct file (~256 bytes) plus inode/dentry cache entries.

PgBouncer — Why It Exists

PgBouncer is a lightweight connection multiplexer. 5,000 app connections hit PgBouncer, which maintains only ~100–200 actual backend connections.

5,000 app connections │ [PgBouncer] ─── transaction-level pooling │ ~150 PostgreSQL backends

OpenAI runs 1 PgBouncer container per replica in Kubernetes — each pod gets its own pooler as a sidecar.

The irony: PgBouncer can't handle prepared statements in transaction mode (the most efficient pooling mode), because prepared statements are tied to a specific backend's session. PgBouncer 1.21+ added prepared_statement_cache to work around this, but it's still a leaky abstraction.

DDL & Schema Changes

What's fast (metadata-only):

• ADD COLUMN with no default or constant default — since PG 11, no rewrite. Lock held for milliseconds.
• DROP COLUMN — metadata-only (marks invisible, doesn't remove data).
• RENAME COLUMN / RENAME TABLE — instant.

What requires a full table rewrite:

• ALTER COLUMN TYPE (e.g., int → bigint) — rewrites every row.
• ALTER COLUMN SET NOT NULL — full table scan (PG 12+ can skip if a matching CHECK constraint exists).
• Adding a column with a volatile default (e.g., DEFAULT now()) — rewrite.

The Real Killer: Lock Queuing

Even for "fast" DDL, PostgreSQL acquires an ACCESS EXCLUSIVE lock. If there's a long-running transaction, the DDL waits. While it waits, every new query queues behind the DDL. This cascading queue causes outages at scale — not the rewrite itself, but the lock contention.

OpenAI added a 5-second lock_timeout — if the lock can't be acquired in 5s, the DDL fails rather than causing a traffic jam.

Community Workarounds

pg_repack, pgroll, pg_osc — "shadow table" approaches (create new table, copy data with triggers, swap) to avoid holding the lock during a rewrite. The fact that these tools need to exist tells the story.

VACUUM — Dead Tuple Problem

PostgreSQL's MVCC stores dead tuples in-line in the heap file. Every UPDATE creates a new tuple version, leaving the old one dead. Every DELETE marks a tuple dead but doesn't remove it. Dead tuples accumulate, file grows without bound.

Regular VACUUM (run by autovacuum) marks dead space reusable but cannot shrink the physical file. A table that grew to 200GB stays at 200GB on disk forever, even with only 50GB of live data.

VACUUM FULL — The Nuclear Option

VACUUM FULL users; -- 100GB table, 50% bloat 1. Takes ACCESS EXCLUSIVE lock → blocks ALL reads AND writes 2. Creates a brand new file 3. Copies every live row (compacted) 4. Rebuilds every index 5. Swaps files, releases lock Duration: hours · Disk: ~2× table size · Impact: total outage for that table

Alternative: pg_repack — rewrites online with only a brief exclusive lock at the end.

Foreign Keys ≠ Indexes

When you declare a foreign key in PostgreSQL, no index is created on the referencing column:

CREATE TABLE orders ( id SERIAL PRIMARY KEY, user_id INT REFERENCES users(id) -- constraint only, NO index );

You must manually create it. Without it: JOINs do sequential scans, DELETEs seq-scan dependent tables. Many production databases have this footgun. Diagnostic query:

SELECT c.conrelid::regclass AS table_name, a.attname AS fk_column FROM pg_constraint c JOIN pg_attribute a ON a.attrelid = c.conrelid AND a.attnum = ANY(c.conkey) WHERE c.contype = 'f' AND NOT EXISTS ( SELECT 1 FROM pg_index i WHERE i.indrelid = c.conrelid AND a.attnum = ANY(i.indkey) );

Index Builds

Default CREATE INDEX: Takes ACCESS EXCLUSIVE lock — blocks ALL reads and writes for the entire build. On a 47M row table, can be minutes to hours.

CREATE INDEX CONCURRENTLY: No write lock, but takes 2× longer, can fail and leave an invalid index, cannot run inside a transaction. Needs to wait for all pre-existing transactions to complete.

Each replica builds the index independently via WAL replay, generating heavy WAL traffic during the build.

ORM-Driven Index Sprawl & Write Amplification

ORMs like Rails ActiveRecord, Django, and SQLAlchemy auto-create indexes in migrations — every FK, every unique=True. A mature Rails app can have 200+ indexes.

Table: users (47M rows), 6 indexes → a single INSERT causes 7 write operations (1 heap + 6 indexes) → each generates WAL → replicated to 50 replicas → 350 total write ops per INSERT

An index that adds +20% write amplification across 50 replicas is catastrophically expensive even if it's never used for reads.

MongoDB Internals

Thread-per-connection · WiredTiger storage · Logical oplog replication

Oplog — Logical Replication

The oplog records the logical operation:

{ "op": "u", "ns": "mydb.users", "o2": { "_id": ObjectId("...") }, "o": { "$set": { "name": "Alice" } } }

~150 bytes. It doesn't care about which pages were touched on disk, index updates (secondaries apply the operation and their own storage engine handles indexes locally), internal page splits, or how WiredTiger organized data physically.

Chained Replication — Built-In

Secondaries pull from each other — the primary might only directly feed 2–3 secondaries, and they feed the rest. Primary outbound for the same workload: ~30–60MB/s total vs. PostgreSQL's ~10GB/s.

Topology is self-managing via chainingAllowed (true by default): if a secondary's sync source goes down, it automatically picks a new one. No manual re-pointing, no custom orchestration.

Connections — Thread-Per-Connection, Shared Process

All connections live inside the mongod process as lightweight threads sharing the same address space:

• ~1MB per connection (thread stack) vs. PostgreSQL's 5–50MB per process
• No context switch TLB flush — threads share address space
• No connection pooler needed — built-in thread pool handles it
• 50,000+ connections practical (limited by maxIncomingConnections)

MongoDB 4.4+ added a service executor thread pool — fixed pool of worker threads services incoming connections, further reducing overhead.

File Descriptors — Shared, Not Duplicated

WiredTiger opens each data file once and all threads access it via pread()/pwrite() (thread-safe with explicit offsets). A MongoDB instance with 5,000 connections holds a few hundred FDs total — not a million like PostgreSQL.

Schema Flexibility — No ALTER TABLE Concept

MongoDB's schema is flexible by design (schemaless documents). Adding a new field is just writing a document with the new field. Schema validation changes are metadata operations. No ACCESS EXCLUSIVE locks, no table rewrites, no lock queuing cascade. This is one of the genuine differentiators at scale.

Compact — Rarely Needed

WiredTiger doesn't have the MVCC dead tuple problem. Dead versions are cleaned up by internal cache eviction. Checkpointing (every 60s) and page-level management handle most fragmentation automatically.

db.users.runCommand({ compact: "users" }) 1. WiredTiger reorganizes pages in-place within existing files 2. Blocks writes on collection, but reads continue 3. Releases freed space back to OS Duration: minutes · Disk: minimal extra · Impact: write-only outage

WiredTiger avoids accumulation: no in-line dead tuples, 60s checkpointing consolidates data, page splits/merges handled internally, freed blocks reused immediately.

MongoDB 7.0+ introduced autoCompact — background compaction without manual intervention. You typically only need compact after extreme scenarios like mass-deleting an entire large collection.

Index Builds — Hybrid Since 4.2

1. Lock at start — brief exclusive lock to register the build. Milliseconds.

2. Bulk build phase — reads and writes continue. Scans collection and builds index in a side table.

3. Lock at finish — exclusive lock to commit. Under heavy writes, must drain pending changes. Can be seconds to minutes.

4. Secondaries — each secondary builds locally. Replication is blocked during build. Build on all secondaries simultaneously and you lose read capacity.

Before 4.2 — two bad choices: foreground (locks entire collection) or background (no lock but slower, less efficient B-tree).

Document Model — Why Fewer Indexes Are Needed

In relational databases, data is normalized across many tables. A "user profile" needs 4+ tables and 4+ JOIN indexes:

SELECT u.name, a.street, o.total, p.method FROM users u JOIN addresses a ON a.user_id = u.id JOIN orders o ON o.user_id = u.id JOIN payments p ON p.order_id = o.id;

In MongoDB, that's one document:

{ "_id": 42, "name": "Alice", "address": { "street": "123 Main" }, "orders": [{ "total": 99, "payment": { "method": "card" } }] }

The math: Relational = N−1 JOIN indexes + filter/sort per table. Document = 0 JOIN indexes + only query-driven indexes.

The trade-off: MongoDB pays with data duplication (denormalization). Shared data embedded in multiple documents means multi-document updates when it changes.

No Foreign Keys — By Design

References between documents are just plain fields (user_id: ObjectId("...")). No constraint enforcement, no automatic indexes. You index only what you query on. Referential integrity is the application's responsibility.

This means zero FK-related index overhead — no forgotten unindexed FKs (PostgreSQL's footgun) and no auto-created FK indexes adding write amplification (MySQL's overhead).

MySQL / InnoDB

Clustered index architecture · Automatic FK indexes · 2-hop secondary lookups

Clustered Index — The PK IS the Table

InnoDB stores data sorted by primary key in B-Tree leaf pages. There is no separate heap. The primary key B-Tree leaf pages contain the full row data. This means:

• Primary key lookups are always fast — you land directly on the data.
• Range scans on the PK are sequential I/O — physically adjacent rows are stored together.
• But random inserts (e.g., UUID PKs) cause page splits and fragmentation.

Secondary Indexes — The 2-Hop Tax

Secondary indexes in InnoDB store (key → primary key value), not a physical address. Every secondary index lookup requires 2 hops:

1. Secondary index B-Tree → find the primary key value
2. Clustered index B-Tree → find the actual row data

Contrast with PostgreSQL/MongoDB where secondary indexes point to a physical address (ctid / RecordId) — always 1 hop.

The 2-hop tax matters at scale: covering indexes (that include all queried columns in the index itself) can avoid the second hop, but they increase index size and write amplification.

Automatic FK Indexes

InnoDB automatically creates an index on every foreign key column — you don't have a choice. This is safer by default (avoids PostgreSQL's unindexed FK footgun) but contributes to index sprawl and write amplification on every INSERT/UPDATE/DELETE, whether you query on those columns or not.

Dead Row Cleanup — All Four Engines

Every database must deal with old row versions created by updates and deletes. Where those dead versions live determines how painful cleanup is:

EngineDead row locationCleanup mechanismManual compactionLock type
PostgreSQL In the heap (in-place) VACUUM (manual / autovacuum) VACUUM FULL ACCESS EXCLUSIVE (total outage)
MongoDB None (WiredTiger cleans internally) Automatic (cache eviction + checkpoint) compact (rarely needed) Blocks writes only
MySQL / InnoDB Undo tablespace (separate from table) Purge thread (automatic) OPTIMIZE TABLE Online DDL (minimal lock in 5.6+)
Oracle Undo tablespace (separate) Automatic Undo Management (AUM) SHRINK SPACE / MOVE ONLINE Online (12c+) or exclusive

PostgreSQL is the only engine that stores dead row versions in the table itself. Everyone else (MongoDB, MySQL, Oracle) keeps old versions elsewhere or cleans them up internally.

MySQL / InnoDB — Purge Thread + OPTIMIZE TABLE

InnoDB avoids the dead-tuple-in-the-table problem entirely:

• Undo log + purge thread: When you UPDATE or DELETE, the old row version goes into the undo tablespace (not the table). A background purge thread asynchronously cleans up undo records once no active transaction needs them. Fully automatic.

• OPTIMIZE TABLE: The closest equivalent to VACUUM FULL. Rebuilds the clustered index by creating a new copy, then swaps. Used when you have significant fragmentation from bulk deletes.

• ALTER TABLE ... ENGINE=InnoDB: A common trick — does the same thing as OPTIMIZE TABLE but works more reliably with online DDL in MySQL 5.6+.

• Page-level reuse: InnoDB pages with deleted rows have free space that gets reused by future inserts. The file doesn't grow without bound the way PG's heap does.

Oracle — Automatic Undo Management

• Undo tablespace: Like InnoDB, old row versions go to a separate undo tablespace. Automatic Undo Management (AUM) handles cleanup.

• ALTER TABLE ... SHRINK SPACE: Compacts the table in-place, reclaiming space from deleted rows. Requires row movement enabled. Can run online (no exclusive lock) but invalidates rowids.

• ALTER TABLE ... MOVE ONLINE (12c+): Full table rebuild without locking. Before 12c, MOVE required exclusive lock and invalidated all indexes (must rebuild after).

• Segment Advisor: Built-in advisor proactively identifies tables with wasted space and recommends shrink operations.

Why VACUUM and compact Exist for Fundamentally Different Reasons

This is a common point of confusion. VACUUM and compact sound similar but solve completely different problems:

PostgreSQL VACUUMMongoDB compact
Why it existsDead tuples accumulate in the table (MVCC design)File-level fragmentation after bulk deletes
What it cleansDead row versions + stale index entriesReorganizes live pages within the file (defragmentation)
Is it mandatory?Yes — without it, table grows forever and eventually transaction ID wraparound shuts down the databaseNo — WiredTiger works fine without it, just wastes disk space
What if you never run it?Database eventually stops accepting writesNothing breaks — you just use more disk than needed
Frequencyautovacuum runs constantly (every few seconds)Maybe once a year after a major data cleanup, if ever
AnalogyTaking out the trash — mandatory or the house becomes uninhabitableDefragmenting a hard drive — nice to have, not necessary

Why MongoDB Doesn't Have the Dead Tuple Problem

WiredTiger has zero dead tuple accumulation. Old document versions are cleaned up by internal cache eviction and 60-second checkpointing. No equivalent of MVCC bloat.

But the physical file on disk can become fragmented after extreme events. Example: a 100GB collection where you delete 80% of documents. WiredTiger immediately reuses freed space within pages (future inserts fill gaps), but the file stays 100GB on disk. The OS doesn't get the space back.

This is a file system problem, not a data structure problem. The B-tree and storage pages are clean — there are no dead documents lingering. Pages just have free space WiredTiger can reuse internally. compact calls WT_SESSION::compact() to reorganize pages, move data toward the beginning, and truncate the file tail.

Key insight: PostgreSQL's VACUUM is garbage collection — mandatory cleanup of dead data inside the table. MongoDB's compact is defragmentation — optional reorganization of live data in the file. MySQL and Oracle don't need either because their purge threads handle dead versions automatically and their storage engines reuse space efficiently. PostgreSQL is architecturally unique in requiring an external process to remain functional.

OpenAI Operational Layers

Rate limiting · Query optimization · ORM governance — not database-specific, but critical at their scale

Rate Limiting — Four Layers of Defense

OpenAI implements rate limiting at four distinct layers, each catching what the others miss:

Layer 1 — App-Level: Per-service quotas. Each internal service (ChatGPT, API platform, batch jobs) gets a budget. Rogue services rejected with HTTP 429 before touching the database.

Layer 2 — Pooler-Level (PgBouncer): max_client_conn and max_db_connections. Catches connection storms — 500 new pods spinning up simultaneously get queued, not forked.

Layer 3 — Proxy-Level: Global QPS caps via a reverse proxy. "No matter what, the DB will never see more than X QPS."

Layer 4 — Per-Query-Digest: Queries hashed by pattern; each pattern gets its own rate limit. An expensive 12-table ORM join can be throttled independently without affecting other queries.

Why Four Layers?

• App limits miss rogue services that don't implement the SDK properly
• Pooler limits miss non-connection-based overload (1 connection sending 10K queries/sec)
• Proxy limits are too blunt to differentiate cheap vs. expensive queries
• Query digest limits alone can't stop a connection storm

The retry storm problem: Service A times out → retries → doubles load → Service B times out → retries → quadruples load. A single runaway service at 50K QPS can take down a cluster that handles 200K QPS comfortably. The four layers contain the blast radius at the outermost possible point.

Query Optimization & ORM Governance

Why ORMs generate bad queries: The object-relational impedance mismatch. ORMs map object graphs to relational tables. Traversing user.orders.products.reviews either lazy-loads (N+1 problem) or eager-loads (massive JOINs). ORM SQL is built for correctness, not performance — SELECT *, JOINs based on model relationships, zero visibility into EXPLAIN ANALYZE.

What OpenAI Actually Does

• Query review gates in CI/CD — EXPLAIN on new queries; block PR if cost is too high
• statement_timeout = 5s — kill any query running longer
• Idle connection timeouts — prevent pool exhaustion from leaked ORM connections
• Index audits via pg_stat_user_indexes — find indexes with idx_scan = 0, drop them
• pg_stat_statements — track query patterns by digest to identify top CPU-consuming queries

Key insight: You cannot fully prevent ORM query/index sprawl. The gate catches problems before production, but it's a continuous process — which is exactly why OpenAI treats this as an operational guardrail layer, not a solved problem.
