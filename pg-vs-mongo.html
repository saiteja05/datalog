Deep-Dive Notes

WAL Replication — Why It Chokes the Primary

Physical vs. Logical: Why WAL Is Bigger Than Oplog

WAL records how the disk changed. Oplog records what the user did. That difference is massive at scale.

PostgreSQL WAL — Physical Replication

WAL operates at the storage page level. PostgreSQL stores data in 8KB pages. When you do:

UPDATE users SET name = 'Alice' WHERE id = 42;

WAL doesn't log "set name to Alice for id 42". It logs:

• The full 8KB data page image (or a diff of it) for the heap tuple
• Page changes for every index on that table (B-tree leaf page modifications) — 5 indexes means 5 more page-level changes
• TOAST table changes if the row has large values
• Visibility map updates (for MVCC)
• Free space map updates

A single logical row change can produce tens of kilobytes of WAL because it's recording the physical byte changes across multiple pages.

Why full page images? After a crash, PostgreSQL needs to recover to a consistent state at the page level. It can't just "replay the UPDATE" because it doesn't know if the page was half-written to disk (torn page). So on the first modification of each page after a checkpoint, WAL writes the entire 8KB page (Full Page Write / FPW). Safe, but extremely verbose.

MongoDB Oplog — Logical Replication

The oplog records the logical operation:

{ "op": "u", "ns": "mydb.users", "o2": { "_id": ObjectId("...") }, "o": { "$set": { "name": "Alice" } } }

That's it. ~150 bytes. It doesn't care about which pages were touched on disk, index updates (the secondary applies the operation and its own storage engine handles indexes locally), internal storage engine page splits, or how WiredTiger organized data physically.

At OpenAI's scale with heavy writes, WAL volume can easily be 10–50× larger than an equivalent oplog for the same logical changes.

Fan-Out: Why 50 Replicas Kills the Primary

PostgreSQL: Primary serializes and ships the bloated WAL stream 50 times. If WAL generation is 200MB/s, that's ~10GB/s outbound from the primary. Each wal_sender process copies and pushes full page images.

MongoDB: Primary's oplog is ~10–20MB/s for the same workload. Secondaries pull from each other (chained replication) — the primary might only directly feed 2–3 secondaries, and they feed the rest. Primary outbound: ~30–60MB/s total.

CPU Cost of WAL Senders

Each wal_sender process on the primary:

• Copies WAL data from shared buffers or disk
• Handles the replication protocol (keepalives, feedback, flow control)
• For synchronous replication (to the standby), blocks the committing transaction until the standby acknowledges

At 50 replicas, that's 50 OS processes competing for CPU time on the same machine that's already handling all write workload.

Why Not Logical Replication for Standbys?

PostgreSQL does have logical replication (since PG 10), but it can't replicate DDL (CREATE TABLE, ALTER TABLE), can't replicate sequences or large objects, has higher apply latency (re-executes through the full query engine), and doesn't support failover — you can't promote a logical replica to primary seamlessly.

Physical WAL replication guarantees a byte-identical copy of the primary, which is what you need for reliable failover. That exactness comes at the cost of verbosity.

The fundamental tension: The very mechanism that makes failover reliable (physical WAL) is the one that chokes the primary when you have 50 replicas hanging off it. This is why OpenAI flagged WAL as a scaling concern.

Each Replica = A wal_sender Process on the Primary

Each replica streaming from the primary consumes a wal_sender process — a full forked PostgreSQL backend. These are controlled by max_wal_senders (separate from max_connections), so they don't block user connection slots. But each one still consumes:

• Its own memory allocation (work_mem, process-level buffers)
• CPU time for reading WAL segments and streaming them over the network
• A file descriptor for the replication TCP socket
• Shared buffer access for reading WAL pages from pg_wal

With 50 replicas directly connected, that's 50 wal_sender processes competing with your actual query workload for CPU, memory, and I/O on the same primary machine.

max_wal_senders — The Connection Ceiling

The max_wal_senders GUC controls how many concurrent WAL sender processes can run. It defaults to 10 (since PG 10). With 50 replicas, OpenAI would need it set to at least 50 — likely something like max_wal_senders = 60 to leave headroom for pg_basebackup, backup tools (Barman), and monitoring tools that read WAL.

Each slot pre-allocates shared memory at startup, so even unused slots have a (small) cost. At their scale the memory overhead is negligible — the real cost is the 50 active processes consuming CPU and network.

It's also possible not all 50 replicas connect directly to the primary. Some may already cascade through the hot standby manually, even if the full automated orchestration layer is listed as future work. Nothing in PostgreSQL prevents pointing a replica's primary_conninfo at another standby today — the gap is the automation to manage topology changes, re-pointing, and monitoring at scale.

Cascading Replication — The Solution PostgreSQL Has (but OpenAI Hasn't Deployed Yet)

PostgreSQL has supported cascading replication since PG 9.2. Instead of every replica connecting directly to the primary, you can chain them:

┌─ Replica 3 ┌─ Standby A ─┼─ Replica 4 │ └─ Replica 5 Primary ─┤ │ ┌─ Replica 6 └─ Standby B ─┼─ Replica 7 └─ Replica 8

The primary only feeds 2–3 intermediate standbys, and those standbys feed the rest. This dramatically reduces the primary's fan-out:

TopologyPrimary wal_sendersPrimary outbound (at 200MB/s WAL)
Direct (current)50~10 GB/s
Cascading via 3 standbys3~600 MB/s
Cascading via 5 standbys5~1 GB/s

Why Hasn't OpenAI Deployed Cascading Replication?

Cascading replication adds complexity:

• Replication lag compounds — each hop adds latency. A replica 2 hops away sees changes later than a direct replica.
• Failover of intermediate standbys — if Standby A dies, all its downstream replicas lose their WAL source. You need automation to re-point them.
• Monitoring complexity — you now have a tree topology to track instead of a simple star.
• No built-in orchestration — PostgreSQL provides the mechanism (primary_conninfo pointing to another standby), but not the management layer to handle topology changes automatically.

OpenAI has flagged this as future development. The tooling to manage cascading topology at their scale (automatic re-pointing, lag monitoring per hop, intermediate standby failover) doesn't exist off-the-shelf and needs to be built.

Contrast with MongoDB: MongoDB's replica sets do chained replication by default — secondaries automatically choose the nearest sync source (often another secondary) via the chainingAllowed setting (true by default). The topology is self-managing: if a secondary's sync source goes down, it automatically picks a new one. No manual re-pointing, no custom orchestration needed.

DDL & Schema Changes — Does PostgreSQL Support Live DDL?

What PostgreSQL Does Well (metadata-only, near-instant)

• ADD COLUMN with no default or with a constant default — since PG 11, this is metadata-only, no rewrite. The ACCESS EXCLUSIVE lock is held for milliseconds.
• DROP COLUMN — metadata-only (marks it as invisible, doesn't physically remove data).
• RENAME COLUMN / RENAME TABLE — instant.

What Still Requires a Full Table Rewrite

• ALTER COLUMN TYPE (e.g., changing varchar(50) to text, or int to bigint) — rewrites every single row.
• ALTER COLUMN SET NOT NULL — requires a full table scan to validate (PG 12+ can skip this if a matching CHECK constraint exists).
• Adding a column with a volatile default (e.g., DEFAULT now()) — rewrite.
• CLUSTER / VACUUM FULL — full rewrite.

The Real Killer: Lock Queuing

Even for "fast" DDL, PostgreSQL acquires an ACCESS EXCLUSIVE lock. If there's a long-running transaction (say a 30-minute analytics query), the DDL waits for it. And while it waits, every new query queues behind the DDL. This cascading queue is what causes outages at OpenAI's scale — not necessarily the rewrite itself, but the lock contention.

That's why OpenAI added a 5-second lock_timeout — if the lock can't be acquired in 5s, the DDL fails rather than causing a traffic jam.

Contrast with MongoDB: MongoDB's schema is flexible by design (schemaless documents), so there's no concept of ALTER TABLE. Adding a new field is just writing a document with the new field. Schema validation changes are metadata operations. This is one of the genuine differentiators at scale.

Tools the Community Built to Work Around It

pg_repack, pgroll, pg_osc — these all do "shadow table" approaches (create a new table, copy data with triggers, swap) to avoid holding the lock during the rewrite. The fact that these tools need to exist tells the story.

Connections — Why PostgreSQL Chokes at Scale

The Process-Per-Connection Architecture

PostgreSQL uses a process-per-connection model inherited from its 1990s Postgres95 roots. Every client connection forks a dedicated OS process — the postgres backend. Each backend gets:

• Its own process memory — work_mem (default 4MB, but can be much more per sort/hash), maintenance memory, temp buffers
• A private copy of the catalog cache — parsed table schemas, type info, function bodies
• A PGPROC slot in shared memory — tracked in the ProcArray for MVCC visibility checks
• An OS-level process — context switching, TLB flushes, scheduling overhead

At 100 connections, this is fine. At 5,000 connections (OpenAI's scale), you have 5,000 OS processes, each consuming 5–50MB of private memory, and the OS scheduler falls apart trying to context-switch between them.

Why High Connection Counts Kill Performance

1. Lock contention on ProcArray: Every transaction start/end must scan the ProcArray (list of all active backends) for MVCC snapshot generation. At 5,000 connections, this scan becomes a serialization bottleneck — each GetSnapshotData() call iterates all 5,000 slots while holding a shared lock.

2. Context switching overhead: The Linux kernel wasn't designed to efficiently schedule 5,000+ CPU-bound processes. At this scale, the scheduler itself burns significant CPU just deciding who runs next. Cache lines (L1/L2/L3) get thrashed on every context switch.

3. Memory bloat: 5,000 backends × 10MB each = 50GB just for backend private memory, before you even count shared_buffers. With work_mem = 64MB (common for analytics), a single complex query per connection could need 320GB.

4. Shared buffer contention: All backends access the same shared buffer pool. Each buffer page has a lightweight lock. At 5,000 backends, hot pages (like a frequently-read index root) become severe lock contention points.

5. File descriptor explosion: Each backend opens its own file handles — PostgreSQL doesn't share FDs across processes. A query touching a table with 5 indexes across 3 partitions needs 18+ file descriptors in that single backend. PostgreSQL caches these per-backend (controlled by max_files_per_process, default 1000) to avoid constant open()/close() syscalls.

At 5,000 backends caching ~200 FDs each, that's ~1,000,000 open file descriptors system-wide. This requires:

• ulimit -n raised to 10,000+ per process (Linux default is 1024)
• Kernel fs.file-max raised to millions
• fs.nr_open raised from its default

Every open FD consumes a kernel struct file (~256 bytes) plus inode/dentry cache entries. At a million FDs, the kernel's VFS layer itself becomes a contention point — file descriptor table lookups, inode cache pressure, and dentry slab allocation all degrade.

Contrast with MongoDB: Since all connections share a single mongod process, file descriptors are shared too. WiredTiger opens each data file once and all threads access it via pread()/pwrite() (thread-safe with explicit offsets). A MongoDB instance with 5,000 connections and the same schema holds a few hundred FDs total — not a million.

Why PgBouncer Exists

PgBouncer is a lightweight connection multiplexer that sits between your application and PostgreSQL. 5,000 app connections hit PgBouncer, which maintains only ~100–200 actual backend connections to PostgreSQL.

5,000 app connections │ [PgBouncer] ─── transaction-level pooling │ ~150 PostgreSQL backends

It works because most connections are idle most of the time — waiting for the app to build the next query. PgBouncer only assigns a real backend for the duration of a transaction, then returns it to the pool.

OpenAI runs 1 PgBouncer container per replica in Kubernetes — each pod gets its own pooler as a sidecar, eliminating the pooler itself as a single point of failure.

The irony: PgBouncer can't handle prepared statements in transaction mode (the most efficient pooling mode), because prepared statements are tied to a specific backend's session. This forces apps to either avoid prepared statements or use session-level pooling (which defeats the purpose). PgBouncer 1.21+ added prepared_statement_cache to work around this, but it's still a leaky abstraction.

Contrast with MongoDB

MongoDB uses a thread-per-connection model with a shared-process architecture. All connections live inside the mongod process as lightweight threads sharing the same address space.

The bottom line: PostgreSQL needs an external component (PgBouncer) to handle what MongoDB does natively. At OpenAI's scale, they run PgBouncer as a sidecar in every pod — that's hundreds of extra containers just to work around a 30-year-old architectural decision. The connection overhead alone consumed engineering months that could have been spent on product.

Rate Limiting — Four Layers of Defense

OpenAI implements rate limiting at four distinct layers, each catching what the others miss. This is defense in depth — the same principle as the entire solar system architecture.

Layer 1: App-Level Rate Limiting

The application itself enforces per-service quotas. Each internal service (ChatGPT, API platform, batch jobs) gets a budget — e.g., "this service can send max 5K QPS to the database." If a service goes rogue (bugs, infinite retry loops, misconfigured batch jobs), it gets rejected with HTTP 429 before the request ever leaves the app layer. This is the first and cheapest line of defense.

Layer 2: Pooler-Level Limiting (PgBouncer)

PgBouncer enforces max_client_conn and max_db_connections. Even if a service bypasses app-level limits, PgBouncer will queue or reject connections that exceed the pool size. This catches connection storms — if a deployment spins up 500 new pods simultaneously, each trying to open connections, PgBouncer absorbs the burst instead of letting all 500 fork PostgreSQL backends.

Layer 3: Proxy-Level Limiting

A reverse proxy sitting in front of the database cluster enforces global QPS caps. This is the aggregate view — regardless of which service is sending traffic, the proxy ensures total throughput to PostgreSQL doesn't exceed what the cluster can handle. It's the "no matter what, the DB will never see more than X QPS" guarantee.

Layer 4: Per-Query-Digest Caps

The most granular layer. Queries are hashed by their pattern (the "digest" — SELECT * FROM users WHERE id = ? regardless of the actual id value), and each pattern gets its own rate limit. This catches the scenario where a single expensive query pattern (like a 12-table ORM join) starts dominating — even if total QPS is under the global cap, that one pattern can be throttled independently without affecting other queries.

Why Four Layers Instead of One?

Each layer catches what the others miss:

• App limits miss rogue services that don't implement the SDK properly
• Pooler limits miss non-connection-based overload (e.g., one connection sending 10K queries/sec)
• Proxy limits are too blunt to differentiate between cheap and expensive queries
• Query digest limits alone can't stop a connection storm

The retry storm problem: Without rate limits, a timeout cascade is inevitable. Service A times out → retries → doubles load → Service B starts timing out → retries → quadruples load. This exponential amplification is why a single runaway service at 50K QPS can take down a cluster that normally handles 200K QPS comfortably. The four layers ensure the blast radius is contained at the outermost possible point.

Query Optimization & Index Management — The ORM Problem

Why ORMs Generate Bad Queries

The object-relational impedance mismatch is the root cause. ORMs map object graphs to relational tables. When application code traverses user.orders.products.reviews, the ORM either lazy-loads (N+1 problem — one query per object) or eager-loads (massive JOINs across every related table). It doesn't understand data distribution, access patterns, or what indexes exist.

ORM-generated SQL is built for correctness, not performance. It uses SELECT * instead of projecting only needed columns. It generates JOINs based on model relationships, not on selectivity or index coverage. It has zero visibility into EXPLAIN ANALYZE — it doesn't know that a 12-table JOIN produces a plan with nested-loop sequential scans touching 4 million rows.

The Index Sprawl Problem

ORMs like Rails ActiveRecord, Django, and SQLAlchemy (via Alembic) auto-create indexes in database migrations — every foreign key gets one, every unique=True field, and sometimes indexes for common lookup patterns. Over time this accumulates. A mature Rails app can easily have 200+ indexes, many unused or duplicated.

Every index on a table must be updated on every INSERT, UPDATE, and DELETE. This is the write amplification problem:

Table: users (47M rows) Indexes: idx_users_pkey → updated every INSERT idx_users_email_lower → updated every INSERT + UPDATE on email idx_users_created_at → updated every INSERT idx_users_status → updated every INSERT + UPDATE on status idx_users_org_id → updated every INSERT + UPDATE on org_id idx_users_name_gin → updated every INSERT + UPDATE on name (GIN = expensive) 6 indexes → a single INSERT causes 7 write operations (1 heap + 6 indexes) Each generates WAL → replicated to 50 replicas → 350 total write ops per INSERT

At OpenAI's scale (millions of writes/day), unnecessary indexes translate directly into: higher WAL volume, more replication lag, more checkpoint I/O, and more disk usage. An index that adds +20% write amplification across 50 replicas is catastrophically expensive even if it's never used for reads.

What OpenAI Actually Does

Complete prevention is impossible — developers will always write new ORM code generating new patterns. Instead, they treat it as a continuous operational layer:

• Query review gates in CI/CD — hooks that run EXPLAIN on new queries against production-like data. If cost exceeds a threshold or a sequential scan appears on a large table, the PR is blocked before merge.
• statement_timeout = 5s — PostgreSQL kills any query running longer than this. The bad 47s ORM join never finishes.
• Idle connection timeouts — kill connections sitting idle too long, preventing pool exhaustion from leaked ORM connections.
• Index audits via pg_stat_user_indexes — periodic review to find indexes with idx_scan = 0 (never used for reads). Drop them to reclaim write bandwidth.
• pg_stat_statements — tracks query patterns by digest to identify top CPU-consuming queries for rewriting or targeted index creation.

Key insight: You cannot fully prevent ORM query/index sprawl. Developers will always write new code. The gate catches problems before production, but it's a continuous process — which is exactly why OpenAI treats this as an operational guardrail layer, not a solved problem.

Index Builds — PostgreSQL vs MongoDB

When ORM-created indexes need to be built (or dropped and rebuilt), the build process itself is a major operational concern:

AspectPostgreSQLMongoDB
Default CREATE INDEX Takes ACCESS EXCLUSIVE lock on the table — blocks ALL reads and writes for the entire duration of the build. On a 47M row table, this can be minutes to hours. Hybrid build (since 4.2) — brief exclusive lock at start and finish, but reads/writes continue during the bulk phase. Much less disruptive.
Concurrent option CREATE INDEX CONCURRENTLY — no write lock, but takes 2× longer, can fail and leave an invalid index, and cannot run inside a transaction. N/A — hybrid is the only mode since 4.2. No choice needed, but also no way to force a faster exclusive build.
Impact on replicas Each replica must build the index independently (physical replication replays the WAL, which includes the index build). Heavy WAL traffic during build. Built on primary, then blocks replication on each secondary while the secondary builds its copy. Secondaries fall behind during build.
The "finish lock" problem CONCURRENTLY needs to wait for all transactions that started before the build to complete — can hang if long-running queries exist. The commit lock at the end must drain the write queue and apply all changes made during the build. Under heavy write traffic, this lock can be significant.
Write amplification Every index adds WAL overhead per write. Replicated to all replicas via WAL shipping. Every index adds oplog overhead per write. Replicated to all secondaries via oplog.
Background builds on secondaries Not applicable — WAL replay is the mechanism. Secondary is read-only. Secondaries build indexes in the foreground, blocking replication. If all secondaries build simultaneously, you lose read capacity.

MongoDB Index Builds — The Nuance

MongoDB's hybrid index build (4.2+) is often described as "background" but this is misleading:

1. Lock at start — brief exclusive lock to register the build in the catalog. Usually milliseconds.

2. Bulk build phase — reads and writes continue. The build scans the collection and builds the index in a side table. This is the "background" part.

3. Lock at finish — this is the important one. The build takes an exclusive lock to commit the completed index. Under heavy write traffic, it must drain pending writes and apply all changes that happened during the build. This lock can be seconds to minutes on write-heavy collections.

4. Secondaries — each secondary receives the index build via the oplog and builds it locally. During this build, replication is blocked. If you're building a large index, secondaries accumulate replication lag. Build on all secondaries simultaneously and you lose read capacity.

Before 4.2 — you had two bad choices: foreground (locks the entire collection, similar to PostgreSQL's default) or background (no lock but slower, more memory, and a less efficient B-tree). Neither was great.

Bottom line: MongoDB's index builds are genuinely less disruptive than PostgreSQL's default CREATE INDEX. But the ongoing write amplification cost of maintaining indexes is identical in both systems. The ORM index sprawl problem — 6 unnecessary indexes causing +119% write overhead — would hurt MongoDB just as much on the write path. The difference is that creating those indexes is less of an outage event in MongoDB, while maintaining them is equally expensive everywhere.

VACUUM FULL vs Compact — Reclaiming Disk Space

The Problem: Dead Space

Both databases accumulate wasted space over time after heavy deletes and updates. How they reclaim it is architecturally different — and the operational impact is dramatically different.

PostgreSQL — VACUUM FULL

PostgreSQL's MVCC stores dead tuples in-line in the heap file. When you UPDATE a row, the old version stays in the file as a dead tuple. When you DELETE, the row is marked dead but remains physically present. Regular VACUUM marks this space as reusable for future inserts, but the file never shrinks.

If you delete 50% of a 100GB table, you still have a 100GB file with 50GB of holes. Regular VACUUM makes those holes fillable, but the file stays 100GB on disk. Only VACUUM FULL physically compacts it.

VACUUM FULL rewrites the entire table to a new physical file:

VACUUM FULL users; -- 100GB table, 50% bloat What happens: 1. Takes ACCESS EXCLUSIVE lock → blocks ALL reads AND writes 2. Creates a brand new file 3. Copies every live row into the new file (compacted) 4. Rebuilds every index on the table 5. Swaps old file for new file 6. Releases lock Duration: hours on large tables Disk needed: ~2× table size (old + new coexist until swap) Impact: total outage for that table for the entire duration

MongoDB — compact

MongoDB with WiredTiger doesn't have the MVCC dead tuple problem. WiredTiger uses document-level concurrency with snapshot isolation, and dead versions are cleaned up by internal cache eviction — not by an external process like VACUUM. WiredTiger's checkpointing (every 60s) and page-level management (automatic splits and merges) handle most fragmentation without any manual intervention.

When you do need to reclaim space (e.g., after a mass delete), the compact command calls WiredTiger's internal WT_SESSION::compact():

db.users.runCommand({ compact: "users" }) What happens: 1. Calls WiredTiger internal compaction 2. Shuffles pages within the existing files — moves data from end toward beginning 3. Blocks writes on the collection, but reads continue 4. Releases freed space at the end of the file back to OS Duration: shorter — no full copy, in-place reorganization Disk needed: minimal extra — no second copy Impact: write-only outage for that collection

Why PostgreSQL Needs VACUUM at All

PostgreSQL's MVCC design is the root cause. Every UPDATE creates a new tuple version and leaves the old one dead in the heap. Every DELETE marks a tuple dead but doesn't remove it. This is by design — it allows concurrent readers to see consistent snapshots without locking. But the cost is that dead tuples accumulate and the table file grows without bound.

Regular VACUUM (run by autovacuum) handles most of this — it marks dead space reusable so future inserts fill the gaps. But it cannot shrink the physical file. A table that grew to 200GB during a spike will stay at 200GB on disk forever, even if only 50GB of live data remains. The only way to reclaim that space is VACUUM FULL (or pg_repack).

Why MongoDB Rarely Needs Compact

WiredTiger's architecture avoids the accumulation problem:

• No in-line dead tuples — old document versions are cleaned up by internal cache eviction, not by an external process
• Checkpointing every 60s — writes consolidated data to disk, naturally compacting
• Page-level management — WiredTiger handles page splits and merges internally, resolving moderate fragmentation automatically
• Block-manager reuse — freed blocks are tracked and reused immediately, unlike PostgreSQL's hole-in-the-middle problem

You typically only need compact after extreme scenarios — like dropping an entire large collection's worth of documents and wanting to return disk space to the OS.

Bottom line: MongoDB compact is significantly less impactful than PostgreSQL VACUUM FULL: it doesn't block reads, doesn't need double the disk space, and is faster. But the more important difference is that you rarely need it at all — WiredTiger self-manages fragmentation, while PostgreSQL's MVCC design guarantees that dead space will accumulate and eventually require manual intervention. The PostgreSQL community knows this is a problem — pg_repack exists specifically to work around VACUUM FULL's outage-inducing lock, and autovacuum tuning is critical at scale. But the need for an external tool to avoid an outage is itself a reflection of the architecture.
