Transactional Models & Guarantees

MongoDB: Multi-Document ACID

MongoDB provides multi-document ACID transactions using a session-based approach. This allows developers to execute a sequence of operations as a single, atomic unit, ensuring data integrity across multiple documents and collections.

// JavaScript Example
const session = client.startSession();
session.startTransaction();
try {
  await accounts.updateOne({ _id: 1 }, { $inc: { balance: -100 } }, { session });
  await history.insertOne({ from: 1, to: 2, amount: 100 }, { session });
  await session.commitTransaction();
} catch (error) {
  await session.abortTransaction();
  throw error;
} finally {
  session.endSession();
}
Key Considerations:
  • Transactions have a default time limit (e.g., 60 seconds).
  • They incur a performance overhead compared to single-document operations.
  • In a sharded cluster, transactions that span multiple shards are more complex and have higher latency.
CockroachDB: Distributed Serializable ACID

CockroachDB provides Serializable isolation by default, the strongest guarantee. Transactions are executed across the distributed cluster, ensuring correctness even with concurrent operations on a global scale.

-- SQL Example
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
INSERT INTO history (from_id, to_id, amount) VALUES (1, 2, 100);
COMMIT;
Key Considerations:
  • The database automatically retries transactions that encounter contention, which the application must handle.
  • For read-modify-write operations, using `SELECT ... FOR UPDATE` is crucial to lock rows and prevent race conditions.
  • Latency is subject to the speed of light; a transaction requiring consensus across regions will be slower than a single-region one.

Indexing Models & Strategies

MongoDB: Flexible Secondary Indexing

MongoDB's strength lies in its ability to create secondary indexes on any field, including fields within arrays and sub-documents. This enables rich, ad-hoc querying.

  • Merits: Extreme query flexibility. Compound indexes can satisfy complex queries efficiently. Multikey indexes automatically index array elements.
  • Limitations: Each index consumes memory and disk space, and slows down write operations. Over-indexing is a common performance pitfall.
CockroachDB: SQL-Standard & Interleaving

CockroachDB uses standard SQL indexing (B-Trees). Its unique feature is `INTERLEAVE IN PARENT` which physically co-locates child rows with parent rows.

  • Merits: Interleaving provides extremely efficient joins for hierarchical data. Standard SQL indexing is familiar and powerful.
  • Limitations: Poor index design (e.g., on a sequential key) can create "hot spots" in a distributed environment, bottlenecking performance.

Transactional Models & Guarantees

MongoDB: Multi-Document ACID

MongoDB's session-based transactions provide full ACID guarantees across multiple documents, which is essential for complex business logic that cannot be contained within a single document update.

// JavaScript Example
const session = client.startSession();
session.startTransaction();
try {
  await products.updateOne({ _id: "item1" }, { $inc: { stock: -1 } }, { session });
  await orders.insertOne({ item: "item1", user: "userA" }, { session });
  await session.commitTransaction();
} catch (error) {
  await session.abortTransaction();
  throw error;
} finally {
  session.endSession();
}
Key Considerations:
  • The document model's ability to embed related data often reduces the need for multi-document transactions in the first place.
  • Transactions add performance overhead and should be used judiciously, not for every write operation.
ScyllaDB: Lightweight Transactions (LWT)

ScyllaDB does not support multi-row ACID transactions. Instead, it offers single-partition atomicity and provides Lightweight Transactions for "compare-and-set" operations using a `IF` clause. This ensures that a write only occurs if a condition is met.

-- CQL Example
UPDATE users SET email = 'new@email.com'
WHERE user_id = 'userA'
IF email = 'old@email.com';
Key Considerations:
  • LWTs are for conditional updates, not for grouping multiple operations into a single atomic unit.
  • They incur a significant performance penalty compared to regular writes because they require a Paxos-based consensus round trip.
  • The lack of multi-row transactions means complex business logic must be handled at the application layer, which can be complex and error-prone.

Indexing Models & Strategies

MongoDB: Flexible Secondary Indexing

MongoDB allows flexible secondary indexes on any field, enabling a wide range of ad-hoc queries beyond the primary key.

  • Merits: Enables rich query patterns. Supports compound, multikey, and geospatial indexes. Easy to add new indexes to support new features.
  • Limitations: Each index has a cost in terms of memory, disk, and write performance. Too many indexes can slow down the database.
ScyllaDB: Primary Key is the Index

In ScyllaDB, the primary key (partition key + optional clustering columns) *is* the index. Queries are highly efficient but must leverage this key.

  • Merits: Extremely fast and predictable query performance. Forces good data modeling practices based on known access patterns.
  • Limitations: Very limited ad-hoc querying. Supporting a new query pattern often requires creating and populating a new table (a form of manual indexing).

MongoDB vs EDB Postgres

The Battle of Greenfield Development vs. Enterprise Modernization

Core Philosophy & Architecture

This comparison is less about competing architectures and more about different strategic goals. MongoDB targets new, agile application development, while EDB provides a powerful, cost-effective path for enterprises to modernize legacy Oracle systems.

MongoDB's Value: Developer Velocity
  • **Flexible Document Model:** Maps directly to application objects, speeding up development.
  • **Schema-on-Read:** Evolve the data model without database downtime or complex migrations.
  • **Rich Ecosystem:** A vast ecosystem built for modern, cloud-native application patterns.
EDB Postgres's Value: Oracle Modernization
  • **Deep Oracle Compatibility:** Native support for PL/SQL, data types, and syntax minimizes rewrite costs.
  • **Risk Reduction:** Provides a proven, low-risk path to migrate off expensive proprietary licenses.
  • **Enterprise-Grade Postgres:** Enhances open-source PostgreSQL with security, performance, and management tools.

Transactional Models & Guarantees

MongoDB:Transactions

MongoDB's transactions provide snapshot isolation and all-or-nothing execution for operations spanning multiple documents, which is crucial for maintaining data integrity in document-centric applications.

// JavaScript Example
const session = client.startSession();
session.startTransaction({
  readConcern: { level: 'snapshot' },
  writeConcern: { w: 'majority' }
});
try {
  // Operations within the transaction...
  await session.commitTransaction();
} catch (error) {
  await session.abortTransaction();
} finally {
  session.endSession();
}
Key Considerations:
  • The document model itself provides atomicity for updates within a single document, which is often sufficient.
  • Developers must explicitly start and manage a session object in their application code to use transactions.
EDB Postgres: Traditional ACID

As an extension of PostgreSQL, EDB offers time-tested, robust ACID transactions with multiple isolation levels. This is the bedrock of relational databases and is ideal for complex, multi-table business logic.

-- SQL Example
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
UPDATE employees SET salary = salary * 1.05 WHERE department_id = 10;
UPDATE department_budgets SET budget = budget - 50000 WHERE id = 10;
COMMIT;
Key Considerations:
  • Developers must be aware of the different isolation levels (`READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE`) and their trade-offs.
  • Long-running transactions can hold locks and block other operations, requiring careful query design.
  • The relational model often necessitates transactions for simple operations that would be atomic single-document updates in MongoDB.

Indexing Models & Strategies

MongoDB: Flexible Secondary Indexing

MongoDB's B-Tree based indexes can be created on any field, providing significant query flexibility.

  • Merits: Supports ad-hoc queries efficiently. Compound and multikey indexes cater to complex document structures.
  • Limitations: High write workloads can be slowed by a large number of indexes. Indexes consume significant RAM.
EDB Postgres: Rich & Specialized Indexing

Postgres offers a wide array of index types beyond the standard B-Tree, each optimized for specific data types and query patterns.

  • Merits: GIN indexes provide excellent performance for `JSONB` documents. GiST indexes are powerful for geospatial data. This specialization can lead to superior performance for specific use cases.
  • Limitations: The variety of index types can increase complexity for developers. Index maintenance (`VACUUM`, `REINDEX`) is a more significant operational concern.

MongoDB vs Amazon DynamoDB

The Flexible Document DB vs. The Cloud-Native Key-Value Store

Architecture Deep Dive

MongoDB: Query Flexibility

Designed for rich, ad-hoc querying. Allows secondary indexes on any field, enabling flexible query patterns without pre-defining every access path.

DynamoDB: Access Pattern Rigidity

Requires you to define your access patterns upfront by creating tables with specific primary keys and Global Secondary Indexes (GSIs). Optimized for speed on known queries.

Transactional Models & Guarantees

MongoDB: Session-Based Transactions

MongoDB's session-based, multi-document ACID transactions provide a familiar and powerful model for developers needing to ensure data integrity across complex operations involving multiple documents.

// JavaScript Example
const session = client.startSession();
session.startTransaction();
try {
  await users.updateOne({ _id: "user1" }, { $set: { status: "premium" } }, { session });
  await subscriptions.insertOne({ userId: "user1", plan: "premium" }, { session });
  await session.commitTransaction();
} catch (error) {
  await session.abortTransaction();
  throw error;
} finally {
  session.endSession();
}
Key Considerations:
  • Offers strong consistency and isolation, making it suitable as a system of record.
  • The document model often reduces the need for transactions as any operation on a single document is atomic.
DynamoDB: TransactWriteItems API

DynamoDB provides ACID transactions for multiple items across one or more tables using the `TransactWriteItems` and `TransactGetItems` API calls. This is a serverless, all-or-nothing operation.

// AWS SDK v3 for JavaScript Example
import { DynamoDBClient, TransactWriteItemsCommand } from "@aws-sdk/client-dynamodb";
const client = new DynamoDBClient({});
const command = new TransactWriteItemsCommand({
  TransactItems: [
    {
      Update: {
        TableName: "Users",
        Key: { "userId": { S: "user1" } },
        UpdateExpression: "SET #st = :s",
        ExpressionAttributeNames: { "#st": "status" },
        ExpressionAttributeValues: { ":s": { S: "premium" } },
      },
    },
    {
      Put: {
        TableName: "Subscriptions",
        Item: { "userId": { S: "user1" }, "plan": { S: "premium" } },
      },
    },
  ],
});
await client.send(command);
Key Considerations:
  • Transactions are limited to a maximum of 100 items.It does not support multi statement transactions.
  • They consume twice the capacity units of a standard write, making them more expensive.
  • The API-based approach can be more verbose than the session-based model in MongoDB.

Indexing Models & Strategies

MongoDB: Flexible Secondary Indexing

MongoDB allows creating secondary indexes on any field, enabling developers to support new query patterns as the application evolves.

  • Merits: High query flexibility. Supports a wide range of index types (compound, multikey, text, geospatial).
  • Limitations: Indexes are not free; they consume RAM and disk space and add overhead to write operations.
DynamoDB: Rigid Primary & Secondary Indexing

DynamoDB's query capabilities are strictly defined by its indexes: the Primary Key, Local Secondary Indexes (LSIs), and Global Secondary Indexes (GSIs).

  • Merits: Extremely fast and predictable performance for queries that use the defined indexes.
  • Limitations: Very little ad-hoc query capability. GSIs are eventually consistent, have their own provisioned capacity (cost), and you are limited to 20 per table. You must know your access patterns in advance.

Scaling from Gigabytes to Terabytes

A practical analysis of the challenges and pitfalls of scaling each database architecture.

MongoDB: The Sharding Journey

10GB to 1TB: The First Shard

At this scale, the primary challenge is deciding when and how to shard. A single replica set can often handle up to 1TB, but performance may degrade. The most critical decision is choosing a shard key. A poor shard key can lead to unbalanced data distribution (some shards hot, others cold) or "jumbo" chunks that cannot be split, creating significant performance bottlenecks. The initial sharding process itself requires downtime or careful planning with a tool like `mongomirror`.

1TB to 10TB+: The Multi-Shard Cluster

At this scale, the system is fully sharded. Challenges become more operational and complex:

  • Shard Key Immutability: Once a collection is sharded, changing the shard key is extremely difficult, often requiring a full data migration. An early mistake becomes a massive future liability.
  • Cross-Shard Queries: Queries that cannot be targeted to a single shard (scatter-gather queries) become increasingly expensive as they must be broadcast to all shards, increasing latency and network traffic.
  • Rebalancing: The cluster's balancer is constantly working to keep data chunks evenly distributed. This process consumes I/O and can impact application performance, requiring careful management of balancing windows.
  • Infrastructure Cost: A 10TB cluster requires numerous shards, config servers, and `mongos` routers, leading to significant infrastructure and management overhead, especially if self-hosted.

CockroachDB: The Seamless Expansion

10GB to 1TB: Adding Nodes

Scaling CockroachDB is operationally simpler. The primary mechanism is adding more nodes to the cluster. The database automatically splits data ranges and rebalances them across the new nodes with no downtime. The main challenge is schema design for performance. Poorly designed primary keys or indexes can create "hot spots" on a single data range, limiting scalability even with more nodes. Understanding data access patterns to avoid transaction contention is key.

1TB to 10TB+: Geo-Distribution & Latency

At a global scale, the challenges shift from capacity to latency and topology:

  • Topology Patterns: Correctly configuring geo-partitioning and leaseholder preferences becomes critical. A misconfigured topology can place data far from its users, negating the benefits of geo-distribution and increasing cross-region network costs.
  • Transaction Latency: While writes are consistent, they are subject to the speed of light. A write that requires consensus from nodes in New York, London, and Tokyo will have a minimum latency dictated by physics. Designing applications to tolerate this is a crucial architectural challenge.
  • "Noisy Neighbor" Issues: In a large, multi-tenant cluster, a single poorly optimized query can consume disproportionate resources. CockroachDB provides observability tools to identify these, but it requires active monitoring.

ScyllaDB: The Linear Scale-Out

10GB to 1TB: Data Modeling is Everything

Like CockroachDB, ScyllaDB scales horizontally by adding nodes to the ring. The process is seamless. The single greatest challenge is data modeling. Because ScyllaDB lacks joins, data must be denormalized for your query patterns. An incorrect data model at 1TB can lead to massive, expensive data migration efforts later. The pitfall is thinking you can "fix the queries later"; in ScyllaDB, the queries dictate the schema.

1TB to 10TB+: Compaction & Operational Tuning

At very large scale, the operational characteristics of the LSM-Tree engine become the primary focus:

  • Compaction Strategy: ScyllaDB is constantly merging SSTables in the background. Choosing the right compaction strategy (Size-Tiered, Leveled, or Time-Windowed) is critical. A wrong choice can lead to massive write amplification (writing the same data many times) or read amplification, impacting performance and SSD endurance.
  • Large Partitions: A single partition that grows too large (e.g., hundreds of MB) can become a bottleneck, causing high latency for reads on that partition. This is a common pitfall that requires careful data modeling to avoid.
  • Repair Operations: Ensuring data consistency across all replicas requires running regular "repair" operations. While ScyllaDB has optimized this process, it still consumes resources and must be managed carefully in a large cluster.

EDB Postgres: The Scale-Up Challenge

10GB to 1TB: The Vertical Climb

Postgres is fundamentally a scale-up architecture. The primary way to handle growth to 1TB is vertical scaling: moving to a larger server with more CPU, more RAM, and faster storage. The main challenges are connection management (using a pooler like PgBouncer is essential) and vacuum tuning. The `autovacuum` process is critical for reclaiming space from dead rows, but at 1TB, default settings are often insufficient, requiring manual tuning to prevent table bloat and performance degradation.

1TB to 10TB+: The Sharding Dilemma

Scaling a single Postgres instance beyond a few terabytes becomes extremely expensive and technically challenging. At this scale, horizontal scaling (sharding) becomes necessary, but it is not a native feature. This presents major pitfalls:

  • Sharding Complexity: Sharding must be implemented either at the application level (a huge undertaking) or using third-party extensions like Citus Data (now part of Microsoft). This adds significant architectural and operational complexity not present in natively distributed databases.
  • Cross-Shard Joins: Performing joins across different shards is either impossible or extremely inefficient, forcing significant application rewrites and data model changes.
  • High Availability: Managing high availability and failover for a sharded Postgres cluster is far more complex than for a single node, requiring sophisticated replication and failover management tools. EDB provides tools for this, but the underlying complexity remains. Postgres is an outstanding database, but scaling it to 10TB+ requires deep expertise and a willingness to manage a complex, multi-faceted infrastructure.

DynamoDB: The Serverless Scale

10GB to 1TB: Partition Key is King

DynamoDB scales by automatically distributing data across partitions based on the hash of the partition key. The primary challenge is designing a high-cardinality partition key that evenly distributes requests. A poor key (e.g., using `status` where most items are `active`) creates a "hot partition," throttling performance for the entire table, as each partition has a fixed capacity limit.

1TB to 10TB+: Cost Management & Query Patterns

At this scale, DynamoDB's fully managed nature handles the operational load, but challenges shift to cost and query flexibility:

  • Cost Optimization: The on-demand model is convenient but can be expensive at high, sustained throughput. Optimizing TCO requires careful capacity planning with Provisioned Capacity and Reserved Capacity, which demands a deep understanding of workload patterns.
  • Query Rigidity: DynamoDB is not a general-purpose query engine. Access patterns are limited to the defined primary key and secondary indexes. Any new query requirement that cannot be served by an existing index often requires creating a new Global Secondary Index (GSI) and backfilling it, or duplicating data into a different table structure.
  • Secondary Index Limitations: While powerful, GSIs are eventually consistent and have their own provisioned throughput, adding another layer of cost and performance management.
